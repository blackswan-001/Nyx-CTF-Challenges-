Fragment C — CVE

File: signal_archive.html · Encoding: Hex → Reverse → Base64 → XOR(0x4E)

4-Layer Decode
8
Open signal_archive.html → View Source → find the SVG element
Open in browser → Ctrl+U. Press Ctrl+F and search for data-enc. The fragment is hidden inside a data-enc attribute on an <svg> element. The SVG is rendered with very low opacity on the page, appearing as a faint repeating pattern.
🔍 Inside the SVG tag — data-enc attribute <svg class="viz" width="400" height="32"
     data-enc="3d6b6e6436706e656a3148662b7833594c675244"
     aria-hidden="true">
There is also a cluster-ref value in SIG-002: 3d6b3364393148666a3148662b7833594c675244. This looks almost identical — a deliberate trap. It decodes to a wrong CVE via the same chain.
9
Understand the encoding: Hex → Reverse → Base64 → XOR(0x4E)
The encoding chain has 4 steps. There are no hints in the file — you must test and identify the pattern. A key observation: converting the hex to ASCII gives a string ending in =knd... — it looks like reversed Base64 (Base64 strings normally end with = padding, not start with it). Reversing the string gives a valid Base64 string. Decoding that then requires XOR with 0x4E (ASCII for 'N' — the first letter of the flag prefix NYX).

CVE-2023-44487
Python — full decode
>>> import base64 >>> raw = bytes.fromhex("3d6b6e6436706e656a3148662b7833594c675244") >>> step1 = raw.decode() # hex → ASCII (reversed b64) >>> step2 = step1[::-1] # reverse the string >>> step3 = base64.b64decode(step2) # decode base64 → raw bytes >>> bytes([b ^ 0x4E for b in step3]) # XOR each byte with 'N' (0x4E) b'CVE-2023-44487'
💡 Where does the XOR key 0x4E come from? Flag prefix = "NYX"
NYX[0] = 'N'
ord('N') = 78 decimal
= 0x4E hex

This is the same key used in Challenge 1 (Shadow Grid).
The pattern: each challenge uses NYX[n] as the XOR key.
🪤 Why SIG-002 cluster-ref is the decoy
!
3d6b3364393148666a3148662b7833594c675244 → same 4-layer decode → CVE-2023-23397 The two hex strings differ by only a few characters. The decoy is placed in SIG-002's body text where it looks like a cross-reference ID, not a fragment. Decoding it yields a real CVE number, making it extra convincing.

Key Lessons from this Challenge

What made it harder than a typical OSINT challenge

i
📖 Techniques used 1. Fragments have no labels — no "fragment_A" tag to find with Ctrl+F.
You must enumerate all suspicious strings in each file.

2. Decoys use the same encoding chain — decoding a string doesn't mean
it's the answer. The decoy for C decodes to a real CVE number.

3. Non-obvious hiding places — SVG data attributes, HTML comment blocks,
and inline text mixed with real data rather than dedicated meta tags.

4. ROT13 as a layer — many players forget ROT13 exists or don't think
to try it as part of a multi-step chain.

5. Reversed Base64 — the string ending with = is the first hint that
something is backwards. This is a classic OSINT/CTF obfuscation technique.

6. XOR key from the flag format — the key 0x4E = 'N' = NYX[0].
This is a meta-clue pattern that rewards players who read the flag format carefully.
