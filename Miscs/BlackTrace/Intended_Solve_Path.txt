Challenge   : Operation AURORA
Category    : OSINT / Phishing
Difficulty  : Hard (500 pts)
Est. Time   : 60-90 minutes

----------------------------------------
CHALLENGE FRAMING
----------------------------------------

Participants are cast as a three-analyst
SOC team tasked with attributing a live
credential-harvesting campaign targeting
financial institutions.

The scenario is framed as a collaborative
exercise — each analyst role is assigned
a different piece of the evidence set.
No single artifact is sufficient alone.
The flag can only be assembled by
cross-referencing all four.

Decoy artifacts, plausible-looking FTP
credentials embedded in email headers,
and a forged kit structure are all present
and designed to slow teams that do not
read methodically.

A successful attribution ties alias,
real identity, city, and campaign tag
across all four evidence sources.
The flag is that attribution.

----------------------------------------
REAL FLAG
----------------------------------------

NYX{blacktrace:arun_kumar:chennai:aurora_v3}

----------------------------------------
INTENDED SOLVE PATH
----------------------------------------

Step 1: ARTIFACT-01 — Email Headers
— Establish the operator alias and campaign tag

  The investigator opens email_headers.txt.
  The raw header dump of the intercepted
  phishing email is immediately readable.

  Key fields:

    FROM      : security-noreply@indusbank-alerts[.]net
    REPLY-TO  : blacktrace_ops@protonmail.com
    X-Campaign: aurora_v3
    X-Operator: blacktrace

  Key findings:
  → Operator self-identifies via X-Operator
  → Campaign tag confirmed via X-Campaign
  → Reply-To leaks operational ProtonMail
  → Sending domain is not affiliated with IndusBank
  → PHPMailer fingerprint links to kit in Artifact 02

  These fields together establish two of the
  four flag components:
    alias      → blacktrace
    campaign   → aurora_v3

  Neither component alone is the flag.
  Both are required for final assembly.

  Note: solvers who attempt to submit the
  ProtonMail address or the sending domain
  as any part of the flag are on the right
  track but misreading which field is the
  operator identifier.

----------------------------------------

Step 2: ARTIFACT-02 — Phishing Kit Analysis
— Extract developer identity from kit forensics

  The investigator opens phishkit_analysis.txt.
  The recovered kit source contains identity
  signals in two locations.

  config.php (key fields):

    $operator   = "blacktrace";
    $campaign   = "aurora_v3";
    $exfil_mail = "arun.kumar1991@gmail.com";

  Git blame residue in send.php:

    Author: Arun <arun.kumar1991@gmail.com>
    Comment: "fixed smtp auth, tested on live panel – Arun"

  Key findings:
  → config.php exfil_mail is the developer's
    personal Gmail — not a burner
  → Git residue confirms first name: Arun
  → The surname is not present in Artifact 02
    alone — it requires cross-reference with
    Artifact 04 (LinkedIn profile)
  → The Telegram alias in README.txt confirms
    the same operator seen in Artifact 01

  This artifact alone yields a partial identity.
  Full name resolution requires Artifact 04.

  Note: solvers who submit NYX{blacktrace:arun:...}
  have correctly identified the first name but
  have not completed the attribution chain.
  The flag requires the full name in snake_case.

  Admin hint trigger:
  → If a team is stuck after Artifact 02, ask
    whether they have checked the LinkedIn
    artifact for the surname.

----------------------------------------

Step 3: ARTIFACT-03 — C2 Traffic Log
— Geolocate the operator session

  The investigator opens c2_traffic_log.txt.
  The network-captured C2 admin session log
  contains two sections of interest.

  Operator session log:

    [2024-01-17 22:14:08 IST] ADMIN LOGIN  IP=49.207.212[.]88
    [2024-01-18 03:41:19 IST] ADMIN LOGIN  IP=49.207.212[.]88

  IP geolocation block (49.207.212[.]88):

    ISP      : Airtel Broadband
    Country  : India
    State    : Tamil Nadu
    City     : Chennai
    Postal   : 600028
    Timezone : Asia/Kolkata

  Key findings:
  → All operator sessions originate from the
    same residential Airtel IP
  → Full geolocation block is present in the
    log — city is explicit, not inferred
  → Chennai is the third flag component
  → Timezone IST +05:30 consistent across
    all timestamp fields

  This artifact yields one flag component:
    city → chennai

  Note: solvers who geolocate only the
  victim IPs and not the admin session IPs
  will not find the operator location.
  The admin session block is distinct from
  the victim harvest log above it.

----------------------------------------

Step 4: ARTIFACT-04 — LinkedIn Profile
— Confirm full name and cross-reference identity

  The investigator opens the LinkedIn screenshot.
  The archived profile shows a developer
  based in Chennai, Tamil Nadu, India.

  Key fields visible on profile:

    Name    : Arun Kumar
    Location: Chennai, Tamil Nadu, India
    Email   : arun.kumar1991@gmail.com (About section)
    Skills  : PHP, cPanel, web hosting, email marketing

  Key findings:
  → Gmail in About section matches exactly
    the exfil_mail from Artifact 02 config.php
  → Full name resolves the partial identity
    from Artifact 02: Arun → Arun Kumar
  → Location confirms the city extracted
    from the C2 geolocation in Artifact 03
  → PHP and cPanel skills are consistent
    with the kit deployment method

  This artifact closes the attribution:
    real_name → arun_kumar

  The flag can now be assembled.

----------------------------------------

Step 5: ASSEMBLY — Construct the attribution flag
— Four components, one flag

  Component sources:

    alias       → Artifact 01 / 02 (X-Operator, README, Telegram)
    real_name   → Artifact 02 + 04 (Git blame + LinkedIn)
    city        → Artifact 03 (IP geolocation block)
    campaign    → Artifact 01 / 02 (X-Campaign, $campaign)

  Flag assembly:

    NYX{blacktrace:arun_kumar:chennai:aurora_v3}

  Common failure modes:

    ✗ NYX{blacktrace:arun:chennai:aurora_v3}
      (first name only — Artifact 04 not used)

    ✗ NYX{blacktrace:Arun_Kumar:Chennai:aurora_v3}
      (capitalised — flag is lowercase throughout)

    ✗ NYX{blacktrace_ops:arun_kumar:chennai:aurora_v3}
      (ProtonMail handle — not the alias field)

    ✗ NYX{blacktrace:arun_kumar:tamil_nadu:aurora_v3}
      (state instead of city)

    ✗ NYX{blacktrace:arun_kumar:chennai:aurora_v3_}
      (trailing underscore — exact string match)

  Validation is case-sensitive.
  The flag must be exactly:
    NYX{blacktrace:arun_kumar:chennai:aurora_v3}
